1 Introduction
Modern relational database management systems rely on a query optimization compo-
nent that determines the most efficient way to execute a database query. When a user
submits a SQL query, the database system may have multiple possible execution strate-
gies. The query optimizer evaluates these alternatives and selects the plan that minimizes
computational cost. The objective of this project is to design and implement a simplified
cost-based query optimizer capable of selecting efficient query execution plans from a set
of candidate alternatives. The system will demonstrate how relational database systems
internally estimate the cost of query execution and use this information to select opti-
mized execution strategies. The implementation will focus on the key stages of query
optimization, including query plan representation, generation of alternative plans, cost
estimation using statistical information, and selection of the optimal plan.
2 System Overview
The proposed system follows a simplified query optimization pipeline similar to that used
in relational database systems. The system accepts a SQL query as input, constructs an
initial logical execution plan, generates multiple candidate execution plans, estimates the
cost of each plan, and finally selects the most efficient plan. The processing flow of the
system can be summarized as follows:
•Query Input
•Logical Query Plan Generation
•Candidate Plan Enumeration
•Cost Estimation
1•Optimal Plan Selection
•Optimized Plan Output
Each stage will be implemented as a separate module of the system.
3 Supported Query Operations
To demonstrate the working of query optimization techniques, the system will support a
simplified subset of SQL queries involving common relational operations.
3.1 Core Operations
The following relational operations will form the core functionality of the optimizer:
•Table Scan (Sequential Scan)
•Selection (Filtering based on conditions)
•Join Operations
These operations will be used to construct logical query plans that the optimizer will
evaluate.
3.2 Join Strategies
The optimizer will consider different strategies for join operations, such as:
•Nested Loop Join
•Hash Join
•Merge Join (optional)
The optimizer will estimate the cost of each strategy and select the most efficient one
based on table statistics.
4 Data Statistics Model
Cost-based optimization relies on statistical information about the data stored in the
database. In this project, simplified table statistics will be used to estimate the cost of
query execution. These statistics may include:
•Number of tuples in each table
•Attribute selectivity estimates
•Estimated size of intermediate results
The statistics will be manually defined or generated for testing purposes to simulate
realistic database environments.
25 System Architecture
The system will consist of the following conceptual modules:
•Query Interface
•Logical Plan Generator
•Plan Enumerator
•Cost Estimator
•Query Optimizer
•Plan Display Module
The logical plan generator will create an initial query plan from the input query. The plan
enumerator will generate alternative execution plans by exploring different join orders and
operator choices. The cost estimator will compute estimated costs for each candidate
plan. The optimizer will then select the plan with the lowest cost.
6 Output Interface
The primary interface of the system will be a command-line environment where users can
enter queries and observe the generated execution plans. The system will display:
•Initial logical query plan
•Alternative candidate plans
•Estimated cost of each plan
•Selected optimized execution plan
This output will allow users to understand how the optimizer selects efficient query exe-
cution strategies.
7 Technology Stack
The implementation focuses on demonstrating database optimization concepts rather
than building a full production database system. The proposed technology stack includes:
•Programming Language: C++
•Development Environment: Linux-based system
•Data Structures: Trees and graphs for query plan representation
•Version Control: Git and GitHub for collaborative development
Additional libraries may be used for parsing or visualization if required during develop-
ment.

9 Performance Evaluation
To evaluate the effectiveness of the optimizer, experiments will be conducted using queries
over simulated datasets. Possible evaluation criteria include:
•Estimated cost comparison between different execution plans
•Impact of join order on query cost
•Efficiency of different join strategies
These experiments will demonstrate the importance of query optimization in database
systems.
10 Possible Optimization Extensions
Depending on development progress, additional optimization ideas may be explored, such
as:
•Join order optimization
•Predicate pushdown
•Early filtering of tuples
•Limiting the search space of the optimizer
These enhancements may further improve the efficiency of the generated query execution
plans.
11 Expected Outcome
At the completion of this project, the implemented system should be capable of accepting
SQL queries, generating multiple execution plans, estimating their execution costs, and
selecting the most efficient plan. The project will provide a practical demonstration of
how query optimizers operate inside modern relational database systems and highlight
the importance of cost-based optimization in efficient query processing.
